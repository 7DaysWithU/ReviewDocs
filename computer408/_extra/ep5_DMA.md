# DMA

## ✅ 正确区分：DMA 请求（DRQ）、DMA 中断请求（IRQ）

在 DMA（Direct Memory Access）方式中，涉及两类不同的信号/事件：

---

### 1. DMA 请求（DMA Request, DRQ）

- **目的**：通知 DMA 控制器 **有数据要传输**（例如外设准备好一批数据）。
- **触发者**：I/O 设备（如磁盘、网卡）。
- **作用**：DMA 控制器收到 DRQ 后，向 CPU 发出 **总线请求（Bus Request, BR）**。
- **CPU 响应**：释放总线控制权（发出 Bus Grant, BG），DMA 控制器接管总线。
- **随后**：DMA 控制器直接在 **内存与外设之间传输数据**，**无需 CPU 干预**。

> 🔹 这个过程 **不涉及中断**，是 **总线仲裁机制**。

---

### 2. DMA 中断请求（Interrupt Request, IRQ）

- **时机**：**一次完整的 DMA 传输完成之后**（例如整块数据传完）。
- **发起者**：DMA 控制器。
- **目的**：**通知 CPU 数据传输已完成**，请执行后续处理（如更新缓冲区指针、唤醒进程等）。
- **机制**：通过 CPU 的中断引脚发出中断信号，触发 **DMA 中断服务程序（ISR）**。

> 🔹 这才是你所说的“请求后处理程序进行收尾工作”的中断。

---

## 🔄 完整 DMA 传输流程

1. I/O 设备准备就绪（一块`block`数据） → 向 DMA 控制器发 **DMA请求（DRQ）**；
2. DMA 控制器向 CPU 发 **总线请求（BR）**；
3. CPU 完成当前总线周期 → 释放总线，发 **总线授权（BG）**；
4. DMA 控制器获得总线 → **直接传输数据**（内存 ↔ 外设）；
5. 传输完成（如计数器归零）→ DMA 控制器向 CPU 发 **DMA中断请求（IRQ）**；
6. CPU 执行 **DMA 中断服务程序**，进行收尾。

---

## ❌ 常见误解纠正

| 误解 | 正确理解 |
|------|----------|
| “DMA 中断用于请求总线” | ❌ 总线请求靠 **BR/BG 信号**，不是中断 |
| “中断发生在数据传输前” | ❌ 中断只在 **传输完成后** 发生 |
| “每次传一个字都中断” | ❌ 通常按 **整块（block）传输完成后才中断** |

---

## ✅ 总结（精准表述）

> - **DMA 请求（DRQ）**：由外设发起，用于启动 DMA 传输，最终导致 DMA 控制器 **获取总线控制权**（通过总线仲裁，非中断）。  
> - **DMA 中断请求（IRQ）**：由 DMA 控制器在 **数据传输完毕后** 发起，用于通知 CPU 执行后续处理。

因此，你原话中：
> “后者才是请求总线用来传输数据的” —— **这是错误的**。

正确说法应为：
> **“DMA 方式中，‘DMA 请求’（或总线请求）用于获取总线以传输数据；‘DMA 中断请求’用于在传输完成后通知 CPU 进行收尾。”**
